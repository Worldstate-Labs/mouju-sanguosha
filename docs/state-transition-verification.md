# 状态转移验证方案

## 验证对象

服务端规则环境按以下接口验证：

```text
state + legal action + explicit random stream -> next state
```

玩家看到的是完整服务端状态的受限 observation，因此观察投影和动作授权属于独立的隐私验证面。

## 第一阶段：可达状态图探索（已实施）

运行 `npm run test:model`。验证器使用生产规则入口 `createGameV2`、`getLegalActionsV2` 和 `applyGameActionV2`，不复制规则实现。

1. 使用固定 seed 完成 16 局覆盖引导牌局，覆盖 2 人和 4–10 人模式。
2. 从引导轨迹提取新的 `模式/人数/决策/武将/栈顶` 状态签名。
3. 用真实的装备、出牌、响应和伤害动作链补入极少自然轨迹难命中的可达根状态，例如【麒麟弓】。
4. 对每个被处理状态，枚举服务器返回的全部 exact 动作；对模板动作枚举允许数量范围内的全部实体牌组合和目标组合。
5. 对每条转移执行两次并要求结果完全一致，同时确认输入状态没有被修改。
6. 对下一状态检查牌张守恒、实体牌唯一、玩家/座位、体力、存活、回合、胜者和临时区不变量。
7. 未结束状态必须只有一个可行动席位且至少有一个合法动作；结束状态必须有胜者且所有席位均无合法动作。
8. 每个合法动作必须产生语义进展，不能形成原地循环。

默认预算会完整处理 1,000 个去重状态。当前固定结果为 16 局引导、772 个引导根、6,877 条穷举转移、34/34 种 pending、25/25 名行动武将；单状态最多枚举 895 个动作，没有因单状态动作上限而跳过状态。仍有未展开的 frontier，命令会明确输出数量，不能把该结果描述成完整状态空间穷举。

可通过以下环境变量扩大边界：

- `MODEL_TRACE_RUNS`
- `MODEL_MAX_GRAPH_STATES`
- `MODEL_MAX_GRAPH_TRANSITIONS`
- `MODEL_GRAPH_DEPTH`
- `MODEL_MAX_ACTIONS_PER_STATE`

## 第二阶段：独立规则 oracle

为每种牌、技能和规则帧建立独立于 TypeScript handler 的规则契约：前置条件、动作集合、实体牌移动、伤害/回复、触发顺序、公开事件和后置条件。状态图中的每条转移同时交给生产实现与 oracle，比较结构化事件和下一状态。只有独立 oracle 才能发现“实现始终自洽但整体理解错规则”的问题。

## 第三阶段：缩小模型穷举

建立保留花色、颜色、牌类、距离、身份、技能时机的缩小牌堆和武将子集。对缩小模型从初始状态执行 BFS，枚举每个 chance 结果和每个合法动作，配合座位重命名、等价牌归一化和独立动作约简完成闭包。该阶段给出的是缩小模型边界内的完整状态空间结论。

## 第四阶段：形式化状态机

将 pending、frame、死亡、回合移交、胜负和信息投影写成形式化规范。用模型检查验证无死锁、唯一控制者、终局清理和秘密信息非干扰；关键不变量进一步使用归纳证明：初始状态成立，且每类转移均保持，因此任意长度的合法执行都成立。

## 结论口径

第一阶段增强的是“被探索可达状态中的全部合法动作”保证，不是所有完整状态的数学证明。只有独立 oracle、缩小模型闭包和形式化证明逐步完成后，才能对对应边界给出更强的全称结论。
